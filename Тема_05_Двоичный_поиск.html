Мотивация

Зачем нужен двоичный поиск?

Представь, что нужно угадать число от 1 до 100. Какую стратегию выбрать?

❌ Простой поиск (Linear)

Проверяем числа по порядку: 1, 2, 3, 4...

Худший случай: 100 попыток для 100 чисел
Для 1 000 000: до 1 000 000 попыток 😱

Это O(n) — линейное время.

✅ Двоичный поиск (Binary)

Всегда называем середину и отбрасываем половину!

Худший случай: 7 попыток для 100 чисел
Для 1 000 000: всего 20 попыток 🚀

Это O(log n) — логарифмическое время.

⚠️ Важно: двоичный поиск работает только на отсортированных данных!
Теория — из книги «Грокаем алгоритмы»

Как работает двоичный поиск

Ищем число 37 в отсортированном массиве от 1 до 100. На каждом шаге называем середину.

1

Диапазон: [1, 100]. Середина = 50. Получаем ответ: «меньше». Исключаем числа 50–100.

2

Диапазон: [1, 49]. Середина = 25. Получаем ответ: «больше». Исключаем числа 1–25.

3

Диапазон: [26, 49]. Середина = 37. Получаем ответ: «угадал!» 🎉

Ключевая идея: каждый шаг вдвое уменьшает область поиска. Начали со 100 чисел — уже на 3-м шаге нашли! При линейном поиске понадобилось бы 37 шагов.

Алгоритм

Шаблон двоичного поиска

Три обязательных переменных:

L

left = 0 — левая граница поиска

R

right = len(nums) - 1 — правая граница

M

mid = (left + right) // 2 — середина

Цикл продолжается пока left <= right
На каждом шаге сдвигаем либо left, либо right. 
def binary_search(nums, target):
    left = 0
    right = len(nums) - 1

    while left <= right:
        mid = (left + right) // 2

        if nums[mid] == target:
            return mid          # нашли!
        elif nums[mid] < target:
            left = mid + 1      # ищем правее
        else:
            right = mid - 1     # ищем левее

    return -1   # не нашли

Сложность

Почему O(log n)?

На каждом шаге двоичный поиск вдвое уменьшает количество оставшихся элементов. Это и есть логарифм с основанием 2.

log₂(n) — это количество раз, которое нужно поделить n на 2, чтобы получить 1.
n = 1 000 000 → делим пополам

Задача
🟢 LeetCode #704 · Easy

Binary Search

Условие

Дан массив целых чисел nums, отсортированный по возрастанию, и целое число target.

Напиши функцию для поиска target в nums. Если target существует, верни его индекс. Иначе верни -1.

Пример 1:
nums = [-1, 0, 3, 5, 9, 12], target = 9
Ответ: 4
Пример 2:
nums = [-1, 0, 3, 5, 9, 12], target = 2
Ответ: -1
Ограничения: 1 <= n <= 10⁴, все элементы уникальны

Решение

class Solution:
    def search(
        self, nums: list[int], target: int
    ) -> int:
        left = 0
        right = len(nums) - 1

        while left <= right:
            mid = (left + right) // 2

            if nums[mid] == target:
                return mid
            elif nums[mid] < target:
                left = mid + 1
            else:
                right = mid - 1

        return -1
Время
O(log n)
Память
O(1)

Поиск пика

Как найти вершину горы?

Иногда мы ищем не конкретное значение, а пик — элемент, который больше своих соседей. Двоичный поиск работает и здесь!

Ключевое наблюдение:

1

Берём середину mid и смотрим на соседа справа.

2

Если nums[mid] < nums[mid+1] — мы находимся на подъёме, пик правее → сдвигаем left

3

Иначе — мы на спуске или на пике → пик либо здесь, либо левее → сдвигаем right

Это работает, потому что в «горном» массиве всегда есть ровно один пик, и каждое сравнение говорит нам, в какую сторону он находится.

🟢 LeetCode #852 · Easy

Peak Index in a Mountain Array

Условие (на русском)

Назовём массив «горным», если выполняются условия:

  • длина массива >= 3
  • существует такой индекс i, что: arr[0] < arr[1] < ... < arr[i] > arr[i+1] > ... > arr[arr.length-1]

Дан гарантированно горный массив arr. Верни индекс i такой, что arr[i] — наибольший элемент массива.

Пример 1: arr = [0, 1, 0] → 1
Пример 2: arr = [0, 2, 1, 0] → 1
Пример 3: arr = [0, 10, 5, 2] → 1
Ограничения: 3 <= n <= 10⁵, все элементы уникальны

Решение

class Solution:
    def peakIndexInMountainArray(
        self, arr: list[int]
    ) -> int:
        left = 0
        right = len(arr) - 1

        while left < right:
            mid = (left + right) // 2

            if arr[mid] < arr[mid + 1]:
                # мы на подъёме → пик правее
                left = mid + 1
            else:
                # мы на спуске или на пике
                right = mid

        # left == right == индекс пика
        return left
Время
O(log n)
Память
O(1)
⚠️ Цикл: left < right (не <=), потому что мы не ищем конкретный target — мы сужаемся к точке.

Сравнение

#704 vs #852 — в чём разница?

#704 — ищем конкретный target

  • Условие цикла: left <= right
  • На каждом шаге три ветки: ==, <, >
  • При == сразу возвращаем mid
  • Если не нашли — возвращаем -1
while left <= right:
    mid = (left + right) // 2
    if nums[mid] == target:
        return mid
    elif nums[mid] < target:
        left = mid + 1
    else:
        right = mid - 1
return -1

#852 — ищем пик (нет точного target)

  • Условие цикла: left < right
  • На каждом шаге две ветки: вверх / вниз
  • При условии подъёма: left = mid + 1
  • При спуске: right = mid (не mid-1!)
  • Ответ — left после выхода из цикла
while left < right:
    mid = (left + right) // 2
    if arr[mid] < arr[mid + 1]:
        left = mid + 1
    else:
        right = mid
return left
⚠️ Почему right = mid, а не mid - 1? Потому что сам mid может быть пиком! Мы не исключаем его, а сужаем до него.

Итог

Что запомнить

🔑 Шаблон

  • left = 0
  • right = n - 1
  • mid = (left + right) // 2
  • Цикл: while left <= right
  • Три ветки: ==, <, >
  • Не нашли → return -1

🏔 Поиск пика

  • Цикл: while left < right
  • Сравниваем mid и mid+1
  • Подъём → left = mid + 1
  • Спуск → right = mid
  • Ответ: return left

⚡ Сложность

  • Время: O(log n)
  • Память: O(1)
  • Только для отсортированных данных
  • 1 млн элементов → 20 шагов
